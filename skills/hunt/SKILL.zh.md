# Hunt：先诊断，再修复

针对症状打的补丁会在其他地方制造新的 bug。先找到根源。

**在能用一句话说清根本原因之前，不要动代码。**

## 定向

从构建完整的事件图景开始：

- 获取确切的错误信息、调用栈和复现步骤。缺少任何信息时，提出一个具体问题。
- 对错误信息或调用栈中涉及的文件运行 `git log --oneline -20`。如果没有提到具体文件，对整个仓库运行。回归问题几乎总是藏在最近的变更中。
- 从症状向后追踪执行路径：跟着数据走，不靠直觉。
- 自己复现它。如果无法稳定复现，说明还没有真正理解它。

继续之前，先锁定一个可测试的断言：
> "我认为根本原因是 [X]，因为 [证据]。"

断言必须指出具体的文件、函数、行或条件。"一个状态管理问题"不可测试。"因为依赖数组缺少 `userId`，`src/hooks/user.ts:42` 中的 `useUser` 存在过期缓存"才是可测试的。如果还没有这种具体程度，说明还没有假设。

## 已知失败模式

假设难以形成时，将症状与已知模式对照：

| 模式 | 线索 | 排查位置 |
|------|------|----------|
| 时序问题 | 间歇性，依赖负载 | 共享状态的并发访问 |
| 缺少守卫 | 字段或索引访问时崩溃 | 可选值、空集合 |
| 顺序 bug | 单独运行正常，顺序执行失败 | 事件回调、事务范围 |
| 边界失败 | 超时、响应格式错误 | 外部 API、服务边界 |
| 环境不一致 | 本地通过，CI 失败 | 环境变量、功能开关、种子数据 |
| 过期值 | 显示旧数据，重启后刷新 | 内存缓存、memoize 结果 |
| 条件等待 | 用固定时间睡眠代替等待状态 | 将固定 sleep 替换为轮询：检查文件是否存在、端口是否响应、进程是否退出 |

还值得检查：失败位置附近的 TODO，以及这个区域是否已经被打过补丁。同一位置反复修复意味着抽象层出了问题。

注意偏转信号。当开发者或用户说"那部分无所谓"或"不用管那个区域"时，将其视为信号而非许可。一个人不愿审查的区域，往往正是问题所在。

## 检查知识库

深入排查之前，先搜索相同或类似 bug 的历史解决方案：

```bash
ls docs/solutions/ 2>/dev/null && grep -rl "keyword from error" docs/solutions/ | head -5 || true
```

如果找到匹配，读取解决方案文件。其中可能包含根本原因、哪些方法失败了以及修复方式。这可以省掉整个排查过程。

## 检查已知问题

深入排查之前，先检查该 bug 是否已知：

```bash
# 搜索与症状相关的 open issue
gh issue list --search "keyword from error" --state open

# 检查最近是否有相同模式的 CI 失败
gh run list --status failure --limit 5
```

如果未安装 `gh`：`brew install gh && gh auth login`。

## 确认或否定假设

添加一个定向探针：一行日志、一个失败断言，或能在假设正确时失败的最小测试。运行它。

如果证据与假设矛盾，完全丢弃该假设，并以刚刚了解到的内容重新定向。不要保留被证据推翻的假设。

三次假设失败后，停下来。不要猜第四次。而是向用户说明现状：检查了什么，排除了什么，还有什么未知。询问是否要增加更多探针、升级问题，还是以不同方式处理。

**修复后症状相同是硬性停止。** 如果用户报告打补丁后症状依旧，不要再次打补丁。将其作为新的排查来处理：之前的假设是错的。从头重新读执行路径。同一区域出现三轮"修复但仍然有问题"意味着抽象层出了问题，不是某行具体代码的问题。

**永远不要凭记忆陈述环境信息。** 在诊断操作系统、编译器、SDK 或工具版本问题之前，先运行检测命令：`sw_vers`、`xcodebuild -version`、`node --version`、`rustc --version` 等。陈述实际输出。基于假设版本的诊断不算诊断。

**外部工具或 MCP 失败：先诊断，再切换。** 当 MCP 工具、CLI 依赖或外部 API 不可用或返回错误时，不要立即尝试替代方法。先弄清楚为什么失败：服务器在运行吗？API 密钥是否有效或过期？配置是否指向正确的端点？是否需要代理？不诊断根本原因就切换到变通方案，会让原始问题继续存在，并在下次会话中浪费时间。

如果发现自己在做以下事情，停下来重新评估：
- 还没追踪完流程就开始写修复
- 在想"让我试试这个"
- 每次修复都在不同模块中暴露新问题

## 自我合理化监控

这些短语是伪装成诊断的失败。当其中一个出现时，停下来重新审视：

| 你在想什么 | 它实际意味着什么 | 规则 |
|------------|------------------|------|
| "就试这一件事" | 没有假设，随机游走 | 停。先写出假设。 |
| "我确信是 X" | 信心不是证据 | 运行能证明它的探针。 |
| "可能和之前的问题一样" | 将新症状当作已知模式处理 | 从头重新读执行路径。 |
| "在我机器上能跑" | 环境差异本身就是 bug | 在排除之前，列出每一处环境差异。 |
| "再重启一次应该就好了" | 在回避错误信息 | 逐字读取最后一条错误。没有新证据时，不要重启超过两次。 |

## 应用修复

一旦根本原因得到确认：

- 修复原因，而不是它产生的症状
- 保持 diff 尽量小：最少文件，最少行数
- 编写一个回归测试，在未修复的代码上失败，在修复后通过。如果 bug 不可测试（时序问题、环境特定、UI 渲染），说明原因并添加最佳可用的防护。
- 对于大型项目，先运行受影响模块的定向测试集。只有在定向测试通过后才运行完整套件。粘贴完整输出，不作摘要。
- 如果变更涉及超过 5 个文件，暂停并与用户确认范围

**自我调控：** 追踪修复进展。如果同一区域已回滚两次，或当前修复为了一个最初的单个 bug 却涉及超过 3 个文件，停下来。不要继续打补丁。描述已知和未知的内容，并询问用户如何继续。过了这一点还在继续打补丁，意味着抽象层出了问题，而不是某行具体代码。

修复落地后，考虑第二层防御是否合理：在调用点、服务边界或测试中再次验证相同条件。让 bug 无法再次引入，比修复一次更好。

## 常见陷阱

来自历史会话的真实失败，按频率排序：

- **修复了错误的代码路径。** 凭症状猜测位置，在客户端面板而非本地面板打了补丁。在动任何文件之前，从症状向后追踪执行路径。
- **同一症状，四次补丁。** 应用了一个修复，又出现同样的错误，再次打补丁。每次补丁都把真正的原因埋得更深。修复后症状相同 = 停下来，从头重读整个执行路径。
- **凭记忆陈述 macOS 版本。** 把 notarytool 失败诊断为"macOS 26 beta"，实际上是稳定版本。先运行 `sw_vers`。基于假设版本的诊断不算诊断。
- **诊断工具失败之前就尝试变通方案。** xcrawl MCP 没有加载，我试了 WebFetch，而不是先检查原因。在切换方法之前，检查服务器状态、API 密钥有效性和配置。
- **追踪还没完成就写了修复。** "让我试试这个"是一个警示信号。意味着假设还不完整。停下来，完成追踪，用一句话写出假设，再写修复。
- **盲目重启循环。** 重启了服务器 8 次，每次改变一个变量，却没有读取实际的错误响应体。重启之前逐字读取最后一条错误；没有新的基于证据的假设时，不要重启超过两次。
- **流水线健康但无输出。** 编排器报告 RUNNING，但 TTS 供应商配置错误。轮询了状态而没有测试各阶段。在多阶段流水线（ASR->LLM->TTS）中，当编排器说健康但缺少输出时，逐阶段隔离测试。

## 结果

以简短摘要结束：

```
Root cause:  [问题所在，file:line]
Fix:         [变更内容，file:line]
Confirmed:   [证明修复有效的证据或测试]
Tests:       [通过/失败数量，回归测试位置]
```

状态为以下之一：**resolved（已解决）**、**resolved with caveats（已解决，但有注意事项）**（说明注意事项），或 **blocked（受阻）**（说明未知内容及原因）。
