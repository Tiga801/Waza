## 英文写作场景

消除可预判的 AI 写作模式。像人一样写作：多变、不完美、具体。

### 核心规则

1. **删除填充短语。** 去掉开头的铺垫语、强调拐杖词和所有副词。
2. **打破程式化结构。** 避免二元对比、否定排列、戏剧性碎片化、修辞铺垫、虚假施动。
3. **使用主动语态。** 每个句子都需要一个人类主体在做某事。不让无生命物体执行人类动作（"投诉变成了修复"）。
4. **具体化。** 不用模糊的陈述（"原因是结构性的"）。说出具体的事物。不用含糊的极端词（"每个"、"总是"、"从不"）做模糊铺垫。
5. **让读者置身现场。** 不用旁观者视角。"你"胜过"人们"。具体细节胜过抽象概念。
6. **变化节奏。** 混合句子长度。两个例子优于三个。段落结尾要有变化。不用破折号。
7. **信任读者。** 直接陈述事实。省略软化语、解释和搀扶。
8. **删除金句。** 如果一句话听起来像摘录语录，重写它。
9. **不要用另一种程式替换原有程式。** "像人"不等于俚语化、怪诞或刻意随意。

### 词语选择

**滥用副词：全部删除：**
"quietly（悄悄地）"、"deeply（深深地）"、"fundamentally（从根本上）"、"remarkably（显著地）"、"arguably（可以说）"、"certainly（当然）"、"really（真的）"、"just（只是）"、"literally（字面上）"、"genuinely（真诚地）"、"honestly（诚实地）"、"simply（简单地）"、"actually（实际上）"

> 错误："quietly orchestrating workflows" / "fundamentally reshape how we think"
> 正确：说它做了什么。去掉副词。

**AI 词汇：用普通语言替换：**

| 避免 | 替代 |
|------|------|
| delve (into) | examine, look at, explore |
| leverage（作动词） | use |
| utilize | use |
| robust | strong, reliable, solid |
| streamline | simplify, cut |
| harness | use, apply |
| navigate (challenges) | handle, address |
| unpack | explain, examine |
| paradigm | system, approach, model |
| synergy | combination, cooperation |
| ecosystem | community, network, field |
| tapestry | mix, combination |
| landscape | situation, field, area |
| game-changer | significant, important |
| deep dive | analysis, examination |
| moving forward | next, from now |

**浮夸系词：用 "is" 代替：**
> 错误："serves as"、"stands as"、"marks"、"represents"
> 正确："is"

### 需要避免的句式结构

**否定并列**：最常见的 AI 写作特征：
> 错误："It's not bold. It's backwards." / "Not because X, but because Y." / "The question isn't X. The question is Y."
> 正确：直接陈述 Y。完全去掉否定。

**否定倒计时：**
> 错误："Not a bug. Not a feature. A fundamental design flaw."
> 正确："It's a fundamental design flaw."

**修辞自问：**
> 错误："The result? Devastating." / "The worst part? Nobody saw it coming."
> 正确：直接陈述："The result was devastating."

**首语重复滥用**：重复使用相同的句子开头：
> 错误："They assume that... They assume that... They assume that..."
> 正确：合并或重构。一个陈述，清晰的主语。

**三重结构滥用**：连续使用三次"三件事"结构：
> 错误："Products impress people; platforms empower them. Products solve problems; platforms create worlds."
> 正确：把观点说清楚一次。

**虚假范围：**
> 错误："From innovation to cultural transformation"（中间是什么？）
> 正确：直接列出两件事，或选其一。

**戏剧性碎片化**：通过片段制造人为强调：
> 错误："He published this. Openly. In a book. As a priest."
> 正确：完整句子。信任内容本身而非呈现方式。

**虚假施动**：无生命物体执行人类动词：
> 错误："the complaint becomes a fix" / "the decision emerges" / "the data tells us"
> 正确：说出人名。"Someone fixed it." "The team decided."

### 需要避免的语气模式

**虚假悬念过渡：**
> 错误："Here's the kicker." / "Here's the thing." / "Here's where it gets interesting."
> 正确：直接说观点。不要铺垫。

**居高临下的类比：**
> 错误："Think of it like a highway system for data." / "Think of it as a Swiss Army knife."
> 正确：直接解释概念。如果类比有帮助，用三个标准检验后再保留：(1) 去掉它段落就垮了：它是承重的而非装饰的；(2) 再深入一层仍然成立；(3) 读者无需进一步解释就能理解。不满足任何一条，就去掉类比，直接陈述想法。

**未来主义邀请：**
> 错误："Imagine a world where every tool you use has a quiet intelligence behind it..."
> 正确：描述实际存在的或你实际提议的东西。

**虚假脆弱**：表演性的自我意识：
> 错误："And yes, I'm openly in love with the platform model"
> 正确：真正的脆弱是具体的、令人不舒服的。省去这层包装。

**宣布简单而不是证明简单：**
> 错误："The reality is simpler and less flattering" / "History is clear"
> 正确：展示证据。不要在证据之前就宣布结论。

**宏大赌注膨胀：**
> 错误："This will fundamentally reshape how we think about everything." / "will define the next era of computing"
> 正确：说它实际做了什么。

**教学式搀扶：**
> 错误："Let's break this down step by step." / "Let's unpack what this really means."
> 正确：从内容开始，而不是先宣告将要讲内容。

**模糊归因：**
> 错误："Experts argue..." / "Industry reports suggest..." / "Observers have cited..."
> 正确：说出专家名字，附上报告链接，引用原话。如果做不到，说明没有来源。

**发明概念标签**：听起来像分析术语但没有根据的复合标签：
> 错误："the supervision paradox" / "the acceleration trap" / "workload creep"
> 正确：直接描述这件事。不要像已有定义一样给它命名。

### 需要避免的段落与篇章模式

**短促片段段落：**
> 错误："These weren't just products. And the software side matched. Then it professionalised."
> 正确：完整句子。不要断章取义制造戏剧感。

**加粗开头列表**：每个列表项以加粗短语开头：
> 错误："**安全性**：基于环境的配置..." / "**性能**：懒加载..."
> 正确：把列表项写成句子，或去掉加粗。不是每个列表都需要标签。

**分形摘要**："告诉你要讲什么；讲；告诉你讲了什么"：
> 错误："In this section, we'll explore... [3000 words later] ...as we've seen in this section."
> 正确：跳过预告和总结，直接写内容。

**带路标的结论：**
> 错误："In conclusion..." / "To sum up..." / "In summary..."
> 正确：结束。不要宣告自己在结束。

**死亡隐喻**：一个隐喻在全文中用了 10 次：
> 错误："The ecosystem needs ecosystems to build ecosystem value."
> 正确：用一次隐喻，然后继续。

**历史类比堆叠：**
> 错误："Apple didn't build Uber. Facebook didn't build Spotify. Stripe didn't build Shopify."
> 正确：说出观点。一个例子就够了。

**单点稀释**：同一论点用 4000 字以 10 种方式重述：
> 正确：说一次。加证据或继续往前走。

**"尽管存在挑战……"句式：**
> 错误："Despite these challenges, the initiative continues to thrive."
> 正确：要么处理挑战，要么不提它。

**"值得注意" / "有必要提及"：**
> 错误："It's worth noting that this approach has limitations." / "Notably," / "Importantly,"
> 正确：直接说这件事。省去宣告。

### 交稿前快速检查

- 有副词吗？全部删除。
- 有被动语态吗？找到施动者，让他们成为主语。
- 无生命物体在做人类动词？说出人名。
- 句子以 "Here's" 开头？直接说重点。
- 有 "not X, it's Y" 对比？直接陈述 Y。
- 连续三句长度相同？打破其中一句。
- 段落以短促金句结尾？改变一下。
- 有破折号吗？删除。
- 模糊陈述（"影响是显著的"）？说出具体影响。
- 元过渡语（"本文其余部分……"）？删除。让文章自然前进。
- 有列表项以加粗标签开头？重新考虑格式。
- 有 "In conclusion" 或 "To sum up"？删除。

### 评分

在每个维度上打 1-10 分：

| 维度 | 问题 |
|------|------|
| 直接性 | 是陈述还是宣告？ |
| 节奏 | 多变还是单调？ |
| 信任 | 尊重读者智识？ |
| 真实性 | 听起来像人写的吗？ |
| 密度 | 有可删减内容吗？ |

低于 35/50：修改。

---

**底线：多变、不完美、具体。某个单一套路用一次可能没问题。问题在于多个套路同时出现，或同一个套路反复使用。**
