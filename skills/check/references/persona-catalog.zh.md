# 专项审查员激活目录

编排器读取完整 diff，使用判断力（而非关键词匹配）决定激活哪些专项审查员。本目录定义了推理所依据的信号。

## 始终启用（无条件）

基础 /check 技能始终启用。专项审查员是叠加在此之上的。

## 条件性专项审查员

### 安全审查员

**Agent 文件：** `agents/reviewer-security.md`
**激活条件：** Standard 或 Deep 深度

当 diff 涉及以下内容时激活：
- 认证或授权逻辑（中间件、守卫、JWT 处理、session 管理）
- 密码学操作（哈希、签名、加密）
- 信任边界处的输入处理（表单字段、API 请求体、URL 参数）
- 用户可控路径上的文件系统操作
- Shell 或子进程执行
- 第三方凭据或 API 密钥处理
- SQL 查询或原始数据库访问

**不激活的情况：** 纯 UI 变更、配置文件更新、仅测试变更、文档更新。

### 架构审查员

**Agent 文件：** `agents/reviewer-architecture.md`
**激活条件：** Standard 或 Deep 深度

当 diff 满足以下条件时激活：
- 新增了模块、包或服务边界
- 变更了公共 API、导出类型或函数签名
- 引入了原本不存在的跨模块 import
- 跨不同目录修改了超过 10 个文件
- 新增或移除了主要依赖
- 重构了组件间的调用方式

**不激活的情况：** 单文件 bug 修复、测试新增、风格变更、文档更新。

## 对抗性检查（仅 Deep）

无单独 agent。编排器在收集所有发现后，作为额外推理轮次运行此检查。

**激活条件：** 仅 Deep 深度（变更超过 500 行，或存在明确的高风险信号：认证、支付、数据变更、外部 API 集成）。

对抗性检查的问题："如果我想通过这个具体的 diff 攻破这个系统，我会怎么做？"

四个攻击角度：
1. **假设违反** — 这段代码假设什么始终成立（格式、顺序、范围）？不成立时会发生什么？
2. **组合失败** — 这段新代码在并发负载或部分失败时与已有系统交互时，什么会出问题？
3. **级联构造** — 什么样的有效操作序列会导致无效状态？
4. **滥用案例** — 第 1000 个请求、部署期间，或两个用户同时编辑同一资源时会发生什么？

报告对抗性发现时附上置信度评分。低于 0.60 的发现不予提交。
