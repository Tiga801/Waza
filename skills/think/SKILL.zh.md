# Think：先设计验证，再动手构建

将粗略想法转化为清晰的已批准方案，并在编写任何代码之前对架构进行压力测试。

在用户批准设计之前，不编写代码，不搭建脚手架，不进行任何实现。不提供代码片段、伪代码或"仅作示例"的代码。只使用文字和图表。

直接给出观点，避免说："这是一个有趣的思路"、"关于这个问题有很多思考角度"、"你可能需要考虑一下"。明确表明立场，并说明什么样的证据会改变这一立场。

## 阶段 0：评估深度

在读取任何文件之前，根据用户的描述评估任务范围。

| 深度 | 特征 | 工作单元 |
|------|------|----------|
| **轻量** | 单文件、配置变更、简单新增。模糊性低。 | 2-4 |
| **标准** | 多文件功能、集成、中等复杂度。 | 3-6 |
| **深度** | 新系统、横切变更、陌生领域，或高风险。 | 4-8+ |

"工作单元"是一个专注的实现步骤，能产出可测试的结果。

用一行说明深度："深度：轻量 / 标准 / 深度 -- 原因。"

**自动重新分类：** 如果第 1 阶段或第 2 阶段揭示了外部契约面（第三方 API、未文档化的系统行为、导出给调用方的共享类型），则提升一个深度级别并说明原因。

## 阶段 1：理解问题

首先运行 `git log --oneline -10` 并读取 CLAUDE.md（如存在）。然后读取用户提到的文件，或与该想法明显相关的文件（入口点、主要模块）。若不清楚哪些文件相关，先询问。再逐一厘清想法：先明确目的，再明确约束，最后明确成功标准。

**优先检查知识库。** 如果项目存在 `docs/solutions/` 目录，在提出任何方案之前，先搜索相关的历史决策或类似问题：

```bash
ls docs/solutions/ 2>/dev/null && grep -rl "keyword from task" docs/solutions/ | head -5 || true
```

如果找到匹配项，在进入第 2 阶段之前先读取。历史方案可能包含决策、权衡或已知失败模式，可以排除错误起点。

**操作文件系统前先确认工作路径。** 在创建、移动或写入文件之前，用 `pwd` 或 `git rev-parse --show-toplevel` 验证绝对路径。不要假设 `~/project` 和 `~/www/project` 是同一个位置。如果用户提供了相对路径或不明确的路径，询问一次以确认完整的绝对路径。

**在请求凭据之前列出所有依赖。** 如果任务需要 API 密钥、令牌或用户未提及的第三方账户，在请求任何凭据之前，列出每个依赖及其一行说明。不要在实现过程中途才提出凭据请求。

**开始前验证外部工具是否可用。** 如果任务依赖 MCP 服务器、外部 API 或第三方 CLI，预先列出它们并在第一个实现步骤之前确认每个工具可达。依赖不可用工具的方案不算方案。

**在 GitHub 上检查现有工作。** 在设计之前，搜索相关的 issue 和 PR：

```bash
gh issue list --search "feature keyword" --state all --limit 5
gh pr list --search "feature keyword" --state all --limit 5
```

如果未安装 `gh`：`brew install gh && gh auth login`。

质疑这是否是正确的问题：
- 用户实际上希望发生什么？不是所描述的功能，而是他们真正关心的结果。
- 如果什么都不做，会有什么变化？是否有更低成本的路径达到同样的结果？
- 代码库中已有什么可以覆盖部分问题？在提出新代码之前，先将子问题映射到现有代码。
- 这个决策在 12 个月后还成立吗？还是会产生阻力？

## 范围模式

在开始时说明模式：

| 模式 | 时机 | 姿态 |
|------|------|------|
| **expand（扩展）** | 新功能、空白起点 | 推高范围，询问什么能让效果好 10 倍。 |
| **shape（塑形）** | 在现有基础上新增 | 保持基线，一次只提出一个扩展选项。 |
| **hold（保持）** | 错误修复、严格约束 | 范围已锁定，确保正确性。 |
| **cut（裁剪）** | 膨胀过大的方案 | 精简到解决真正问题的最小集合。 |

## 阶段 2：提出方案

提供 2 或 3 个选项，包含权衡与推荐。每个选项：一句话摘要、工作量、风险、两个最强的支持理由和反对理由、所依赖的现有代码。始终包含一个最小化选项和一个架构完整的选项。

对比时问自己：
- 哪些决策难以撤销？在这些问题上放慢速度。
- 什么会导致失败？先从那里开始设计防线。
- 我们明确不构建什么？
- 同样的结果是否可以用更少资源实现：更少的字段、更少的状态、更少的 API？

在提出推荐方案之前：先攻击它。问自己什么会让这个方案失败。如果攻击成立，方案需要变形，应该提出变形后的版本。如果攻击彻底推翻了方案，舍弃它并告知用户原因。

在继续之前获得批准。如果用户拒绝设计，不要从头开始。询问具体哪里不对，将这些约束纳入考量，带着收窄的选项集重新进入第 2 阶段。

## 阶段 3：验证架构

一旦方向获批，在实现开始前检查结构正确性：

**范围。** grep 搜索每个子问题的现有实现。标记任何可推迟的内容。超过 8 个文件或 2 个新服务？明确说明。

**依赖与数据流。** 如果超过 3 个组件相互交换数据，画一张 ASCII 图。寻找循环和隐性耦合。追踪主路径，然后破坏它：空输入、空集合、上游超时、部分失败。

**测试覆盖。** 列出每条有意义的路径：正常路径、错误分支、边界情况。列出缺口，注明文件、断言、测试类型。任何没有复现测试的错误修复都不算完成。

**风险。** 说明每个丢失后会降级系统的组件。是否可以在不接触数据的情况下回滚？技术选型是否足够保守，非标准选择会积累维护成本。

如果某个部分无法根据现有信息进行有意义的评估，明确说明："没有看到 Y，无法评估 X。"不要猜测来填补空缺。

**批准的方案中不允许占位符。** 在用户批准之前，每个步骤都必须是具体的。禁止的模式：TBD、TODO、"稍后实现"、"类似于步骤 N"、"待定"、"按需"。有占位符的方案不是方案，而是稍后再做计划的承诺。

## 阶段 4：信心检验

在移交给实现之前，从三个维度评分信心：

1. **问题理解了吗？** 我能用一句话说明用户实际需要什么以及为什么吗？
2. **方案是最简可行的吗？** 有没有我没考虑到的更低成本路径？
3. **未知项已解决或明确推迟了吗？** 方案中没有隐藏的 TBD。

任何维度偏低：循环回去。(1) 低返回第 1 阶段。(2) 低返回第 2 阶段。(3) 低返回第 3 阶段，明确说明每个未解决的未知项。

三个维度都扎实：在已批准的设计摘要末尾用 2-3 句话说明信心评估，然后停止。不要进入实现。

## 常见陷阱

来自历史会话的真实失败，按频率排序：

- **路径假设错误。** 将文件移动到 `~/project`，但仓库在 `~/www/project`。在第一个文件系统操作之前，始终运行 `pwd` 或 `git rev-parse --show-toplevel`。
- **凭据在构建中途才提出。** 在完成三个实现步骤之后才询问 DashScope API 密钥。在开始之前，列出每个依赖及其需要的一行说明。
- **用户要求执行时却给了分析。** 用户说"帮我做"，却得到了三个选项。"帮我做"、"优化"、"改回去" = 立即执行，不要给选项框架。
- **依赖不可用的工具进行设计。** 在未确认 MCP 服务器是否已加载的情况下，规划了依赖 MCP 的工作流。在第一个设计步骤之前验证外部工具可用性。
- **被拒绝的设计从头重来。** 用户说方向不对，应该询问具体哪里失败了，带着收窄的约束重新进入第 2 阶段，而不是从空白开始。
- **假设区域 API 变体是相同的。** 声网国内版和国际版（Agora）有不同的端点、认证方案和支持的厂商，结果针对错误版本进行了构建。在编写集成代码之前，列出所有区域差异。
- **未经询问就新增了运行时。** 跟随官方文档的 FastAPI 示例进入了一个 Next.js 项目，创建了没有人想要的 Python 后端。将文档示例转换为用户现有技术栈；未经明确批准，不得新增语言或运行时。

## 输出

针对第 3 阶段发现的每个问题：
- 问题描述（1 句话）
- 具体建议（"将 X 移动到 Y，因为 Z"，而不是"考虑重构"）
- 修复规模：小、中、大
- 忽略的风险：低、中、高

**已批准的设计摘要：**
- **构建内容**：这是什么（1 段话）
- **不构建的内容**：明确的范围外列表
- **方案**：选择的选项及其理由
- **关键决策**：3-5 条及推理
- **未知项**：实现过程中需要解决的内容

最后对每个架构部分给出一行状态：清晰、已标记，或已跳过并注明原因。
